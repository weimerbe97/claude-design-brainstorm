§ 01Design principles

Six commitments that hold the flow together.

  1. Rewards is the deterministic floor; codes are the upside.

    Every path through the flow ends with rewards activated — even when codes fail, even when codes never existed. This makes rewards the minimum guaranteed value of every interaction.

  2. Progressive loading matches technical reality.

    Rewards loads in ~100ms on URL match; codes need full DOM (~1-3s). The notification paints with rewards as the hero and enhances with codes when available — never blocking the fast path on the slow one.

  3. Copy gradient matches certainty.

    Codes get hedged language ("ready to try", "3,427 saved today"). Rewards gets specific dollar language ("$3.64 back", "rewards activated"). The honesty calibrates user trust correctly.

  4. Visual weight is inversely proportional to user uncertainty.

    Loud on first paint when the offer is unknown. Loudest at success when celebration matters. Smallest at checkout when payment focus matters. Re-expanded post-purchase to close the loop.

  5. Lifetime Savings is the persistent identity element.

    Visible in the header from State 2a onward. Ticks visibly at every value-delivering moment. This is the brand's strongest narrative device — accumulated value made tangible.

  6. The X means "I don't want to engage at all."

    Visible only in States 1 and 2b, before commitment. Once the user has chosen a path, X disappears and Lifetime Savings takes its place. Never two dismiss paths at once.

Timing rules

  • Codes search timeout: 5 seconds. Beyond this, transition to State 2a.
  • Success state auto-collapse: 4 seconds. Pin remains visible for the session.
  • Green accent line on State 2b codes card: Fade after 2–3 seconds.
  • Edge tab persistence: After X dismissal, edge tab persists for the rest of the session; resets next visit.

Low-volume merchant fallback

If the today-activation count in State 2a's social proof card is below 50, swap to brand-level proof: "12M+ members earn rewards with Capital One Shopping." The flow stays identical; only the content card changes. Threshold and fallback content should be configurable per merchant tier.

Exclusions disclosure rules

  • States 1, 2a, 2b (pre-decision): Inline tooltip link beside the cart-amount line. Dotted underline + info icon. Renders on hover (desktop) or tap (mobile).
  • States 3a, 3b, 4a, 4b (activated): Inline disclosure text above "Continue to checkout." Uses tertiary text color at 10–11px. "Exclusions apply:" prefix bolded for scannability.
  • State 5 (checkout pin): No disclosure — activated states already showed it.
  • State 6 (post-purchase): Pending legal review.

Exclusion text is configurable per merchant via the rewards-API response. Default fallback: "Some product exclusions may apply."

Telemetry priorities

Track state-to-state transitions, not just clicks. The interesting funnel questions:

  • What % of users reaching State 1 reach State 3 (any activation)?
  • What % of users at State 2b try codes vs just activate rewards?
  • What % of code-testing sessions reach State 4a vs 4b?
  • What % of activations reach State 6 (closed loop)?

§ 13Open questions

For legal review
  1. Does the State 1 tooltip-only disclosure satisfy regulatory requirements at the moment of value claim, or does full exclusions text need to be inline?
  2. Should the post-purchase confirmation state (State 6) also include the exclusions disclosure? The order is final at that point, which may change the disclosure standard.
  3. For merchants with exclusions text longer than ~3 lines, do we need a "See full exclusions" overlay pattern?
For data infrastructure
  1. The State 2a social proof count needs a source of truth that updates at least once daily. Can the data team produce this per-merchant?
  2. The low-volume fallback threshold (currently 50) should be validated against real data — what's the actual distribution of daily activations per merchant?
For engineering
  1. Edge tab implementation — is one-session persistence the right window, or should it expire faster / slower?
  2. Do we have telemetry capable of tracking state-to-state transitions, not just clicks?
  3. Pre-engagement scenario: user clicks "Activate rewards" in State 1 before codes finish loading. Does the system gracefully handle in-flight codes search?
For design follow-up
  1. Mobile-specific tooltip behavior — tap to toggle, but what's the dismissal pattern when the user is mid-tooltip?
  2. Should State 2b's X swap to Lifetime Savings if the user has been viewing for 10+ seconds, or stay consistent with State 1?
  3. The "See all my savings" CTA in State 6 — what page does this open? Opportunity to design the bridge into broader platform engagement.
